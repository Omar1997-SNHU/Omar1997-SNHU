Enhancement Two

Algorithms & Data Structures

CO2 CO3 CO4 CO5

Source: original and enhanced files for this category live on the module7 branch of Fullstack-Dev.

1. Artifact Description

Same artifact as Enhancement One: the Travlr Getaways MEAN stack application. In its pre-enhancement state, the API returned all trip documents in insertion order with no sorting, filtering, or pagination, and the Angular front end rendered every trip as a flat list with no way to search or reorder results.

2. Justification & Improvement

This artifact is well suited to this category because trip search, sort, and retrieval performance have concrete, user-facing consequences. The enhancement adds a client-side Map cache (O(1) lookup by trip code), a binary search over a code-sorted array, a linear keyword filter across name, resort, and description, a generic comparison sort handling string, numeric, and date fields, and a price-range filter. On the server, a MongoDB sort/skip/limit pipeline adds server-side pagination, guarded by an allowlist that validates the requested sort field before it reaches .sort().

Client-Side Algorithm — Binary Search & Sort

app_admin/src/app/services/trip-data.ts

// Binary search O(log n): input must be sorted ascending by code
binarySearchByCode(sortedTrips: Trip[], code: string): Trip | null {
  let lo = 0, hi = sortedTrips.length - 1;
  const target = code.toUpperCase();
  while (lo <= hi) {
    const mid = Math.floor((lo + hi) / 2);
    const midCode = sortedTrips[mid].code.toUpperCase();
    if (midCode === target) return sortedTrips[mid];
    if (midCode < target) lo = mid + 1; else hi = mid - 1;
  }
  return null;
}

// Timsort O(n log n): generic comparator for string, numeric, and date fields
sortTrips(trips: Trip[], field: SortField, order: SortOrder = 'asc'): Trip[] {
  return [...trips].sort((a, b) => {
    if (field === 'perPerson') {
      return order === 'asc' ? a.perPerson - b.perPerson : b.perPerson - a.perPerson;
    }
    if (field === 'start') {
      const va = new Date(a.start).getTime();
      const vb = new Date(b.start).getTime();
      return order === 'asc' ? va - vb : vb - va;
    }
    const va = (a[field] as string).toLowerCase();
    const vb = (b[field] as string).toLowerCase();
    const cmp = va < vb ? -1 : va > vb ? 1 : 0;
    return order === 'asc' ? cmp : -cmp;
  });
}

Server-Side Algorithm — Sort Allowlist + Pagination

app_api/controllers/trips.js

// Allowlist: prevents arbitrary field injection into .sort()
const SORTABLE_FIELDS = new Set(['name', 'perPerson', 'start', 'resort', 'length']);

const tripsList = async (req, res) => {
  const { sortBy = 'start', order = 'asc', page = 1, limit = 10, q } = req.query;

  const sortField = SORTABLE_FIELDS.has(sortBy) ? sortBy : 'start';
  const sortDir   = order === 'desc' ? -1 : 1;
  const pageNum   = Math.max(1, parseInt(page, 10) || 1);
  const limitNum  = Math.min(50, Math.max(1, parseInt(limit, 10) || 10));
  const skip      = (pageNum - 1) * limitNum;

  const query = q
    ? { $or: [{ name: { $regex: q, $options: 'i' } }, { resort: { $regex: q, $options: 'i' } }] }
    : {};

  const trips = await Trip.find(query).sort({ [sortField]: sortDir }).skip(skip).limit(limitNum);
  return res.status(200).json(trips);
};

4. Reflection

→ What was the key lesson?

The perPerson sort was the most instructive challenge. The field was originally stored as a string (e.g. "$1,995.00"), so a lexicographic sort produced wrong results — "9800" sorted before "4999" alphabetically but after it numerically. Writing the comparator required stripping non-numeric characters before parsing, which made clear how a schema-level type decision has direct algorithmic consequences three layers away.

→ What was the biggest design decision?

Where to place filtering — client-side (fast, no round-trip, but limited by dataset size) versus server-side (scalable, but adds latency per keystroke). I implemented both layers, which taught me that the right answer depends on dataset size and should be an explicit, documented trade-off rather than an unstated assumption.
