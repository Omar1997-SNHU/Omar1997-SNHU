Introduction

Professional Self-Assessment

Completing the Computer Science program at Southern New Hampshire University, and building this ePortfolio in particular, has clarified what kind of engineer I want to be: someone who can take an existing system, understand why it works the way it does, and make deliberate, well-justified improvements rather than rewrites for their own sake. Every artifact in this portfolio is the same application — Travlr Getaways, a MEAN stack travel booking platform — carried through three distinct rounds of enhancement. That choice was intentional. Showing depth on one system, rather than breadth across three disconnected ones, let me demonstrate how decisions in one layer of an application ripple into the others.

Collaborating in a Team Environment

Throughout the program, courses like CS 250 (Software Development Lifecycle) and IT 315 (Object-Oriented Analysis and Design) required producing design documentation meant for an audience beyond myself — UML diagrams, sprint plans, and peer code reviews. IT 145 (Foundation in Application Development) was where I first practiced writing code with a reader other than myself in mind, a habit that shows up directly in this capstone: every enhanced file in the Travlr Getaways codebase carries JSDoc or TSDoc comments explaining not just what a function does but why it was written that way, so a teammate picking up the code six months from now does not have to reverse-engineer my reasoning. The code review video in this portfolio is itself a collaboration artifact — it is written for a manager or peer audience, not a grader, and walks through existing functionality before any enhancement is proposed, which is the same sequence I would use to bring a real team up to speed before touching shared code.

Communicating with Stakeholders

Technical communication has to flex to its audience. The narratives in this portfolio are written for an audience that understands software broadly but may not know Angular or MongoDB internals — the same skill required when explaining a technical trade-off to a product manager or a client. In CS 320 (Software Testing, Automation, and Quality Assurance), I practiced translating defect reports into plain-language risk statements for non-technical stakeholders, a skill I applied again here when justifying why, for example, increasing password-hashing iterations from 1,000 to 100,000 matters even to someone who has never heard of PBKDF2.

Data Structures and Algorithms

Enhancement Two is the clearest evidence of this strength: a Map-based O(1) cache, a binary search over a sorted index, and a generic comparator-based sort that correctly handles string, numeric, and date fields in a single function. These are not abstract exercises — CS 260 (Data Structures and Algorithms) gave me the vocabulary to reason about Big-O complexity, but applying that reasoning to a real price filter and a real search box, where the wrong data type choice silently produces wrong sort order, is what made the trade-offs concrete rather than theoretical.

Software Engineering and Database

Enhancement One demonstrates software design maturity: migrating a dual-rendering architecture (Express/Handlebars plus a separate Angular SPA) into a single, lazy-loaded Angular front end is the same category of architectural consolidation that production teams perform when technical debt accumulates. Enhancement Three extends that maturity into the database layer — redesigning the Mongoose schema, adding compound indexes, and migrating a price field from String to Number after recognizing that the original type choice was quietly breaking numeric sorting upstream in the Angular layer. DAD 220 (Introduction to SQL) gave me the relational foundation that made the trade-offs of a document-oriented schema easier to reason about by contrast — when to embed versus reference, and why a type decision at the schema level has consequences no relational analogy fully prepared me for. CS 360 (Mobile Architecture and Programming) reinforced the same lesson from a different angle, where data formatting decisions on a constrained client have to be made deliberately rather than inherited from whatever the backend happened to send. Seeing the schema-to-algorithm connection in this capstone — a database decision causing a sorting bug two layers away — was one of the most valuable realizations of the whole program.

Security

A security mindset runs through all three enhancements rather than living in just one. Removing the Handlebars rendering layer in Enhancement One eliminates a server-side template injection surface. The sort allowlist in Enhancement Two's API controller prevents arbitrary field injection into a MongoDB .sort() call. Enhancement Three directly hardens authentication: PBKDF2 iterations were raised to meet current NIST guidance, and an account lockout mechanism was added to resist credential-stuffing attacks. CS 305 (Software Security) gave me the foundation for this kind of thinking — treating every input boundary as a place an adversary might probe — and this capstone is where that foundation became a habit applied without being prompted.

How the Artifacts Fit Together

Each enhancement builds on the architecture left by the one before it. Enhancement One unifies the front end into a single Angular codebase and establishes typed domain models. Enhancement Two adds search, sort, and caching logic on top of that typed foundation, exposing a price-formatting problem that traces back to the original database schema. Enhancement Three resolves that problem at its source by hardening the schema itself and the authentication system around it. The three enhancements that follow are presented in that same order — software engineering and design, algorithms and data structures, and databases — so the architectural story reads the way it was actually built.

What follows: a code review video walking through the original artifact, then three enhancement pages — each containing the original code, the enhanced code, the specific skills demonstrated, and a reflective narrative — followed by a course outcomes summary mapping every enhancement back to the five program outcomes.
