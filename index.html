Enhancement Plan

CATEGORY ONE Software Engineering & Design

Full Angular Migration

The Travlr Getaways application currently maintains two separate rendering paradigms — a customer-facing site built with Express and Handlebars (HBS) templates, and an Angular admin SPA connected to a REST API. The enhancement migrates the entire customer-facing layer to Angular, eliminating server-side template rendering and unifying the codebase.

Angular feature modules with lazy loading (TripListModule, TripDetailModule, HomeModule) replace HBS controllers and partials. TypeScript interfaces are defined for all domain entities. Reactive forms with validators and reusable components (NavbarComponent, TripCardComponent) are added throughout.

// BEFORE: HBS server-renders customer pages // AFTER: Angular renders all pages; Express serves API only { path: '', loadChildren: HomeModule } { path: 'trips', loadChildren: TripListModule } { path: 'trips/:code', loadChildren: TripDetailModule } { path: 'admin', loadChildren: AdminModule }

CATEGORY TWO Algorithms & Data Structures

O(1) Cache & Search Pipeline

Currently, the application calls the API on every navigation and uses Array.find() — an O(n) linear scan — for trip lookups. The enhancement introduces TripCacheService, an RxJS BehaviorSubject-backed in-memory cache with a Map for O(1) keyed access.

A client-side filter pipeline adds real-time search by name, resort, and price using Array.filter() with debounceTime(300). Index-based pagination with Array.slice() provides O(1) page retrieval. The narrative documents explicit trade-off analysis between Map memory cost vs. lookup speed and client-side vs. server-side filtering.

class TripCacheService { _trips$ = new BehaviorSubject<Trip[]>([]) _tripMap = new Map<string, Trip>() // O(1) keyed access getByCode(code) → _tripMap.get(code) // O(1) vs O(n) filter(query, maxPrice) → Array.filter() // no API call } searchInput$.pipe( debounceTime(300), distinctUntilChanged() ).subscribe(r => displayPage(r, pageIndex))

CATEGORY THREE Databases

Schema Hardening & RBAC

The existing MongoDB/Mongoose system validates only basic data types. The enhancement adds comprehensive schema validation (required fields, numeric bounds, enums, regex for tripCode), a role field on the User schema, and role-embedded JWT tokens.

An Express middleware verifyRole is applied to all mutating routes (POST, PUT, DELETE), returning 403 Forbidden for unauthorized access. Unique and compound indexes are added for performance (trips.code, users.email, resort + perPersonCost), and a seed script generates 15+ realistic trip documents.

// Schema hardening TripSchema = { code: { String, required, match: /^[A-Z]{3,5}\d{4}$/ }, perPersonCost: { Number, min: 0, max: 100000 } } UserSchema = { role: { enum: ['admin','read-only','guest'] } } // RBAC — protect mutating routes POST /api/trips → verifyRole('admin'), tripsCreate PUT /api/trips/:code → verifyRole('admin'), tripsUpdateOne GET /api/trips → tripsReadAll // public
