Introduction

Professional Self-Assessment

Completing the Computer Science program at Southern New Hampshire University, and building this ePortfolio in particular, has made things clearer about what kind of engineer I want to be: someone who can take an existing system, figure out why it works the way it does, and make deliberate, well-justified improvements rather than doing a rewrite just because. Every artifact in this portfolio is the same application Travlr Getaways , a MEAN stack travel booking platform, carried through three distinct rounds of enhancement. That was not an accident. Picking one system, instead of showing everything as separate little projects, let me show depth on one place, and also show how decisions in one layer ripple into the others, even if they are not obvious at first glance.

Collaborating in a Team Environment

In CS 255 (System Analysis and Design), I produced a full business requirements document and system design document for a client project—DriverPass, a driving-school platform using UML use case, activity, sequence, and class diagrams to give different stakeholders (students, secretaries, IT administrators) a shared picture of how the system would work, before a single line of code was written. CS 230 (Software Design Document for "Draw It or Lose It") reinforced the same discipline but from another angle: translating a client's platform-migration requirements into technology recommendations they could actually evaluate. It’s about balancing technical specifics against client-facing clarity, like getting the story straight first,then details second. That habit shows up directly in this capstone: every enhanced file in the Travlr Getaways codebase carries JSDoc or TSDoc comments explaining not just what a function does but why it was written that way, so a teammate picking up the code six months from now does not have to reverse-engineer my reasoning. The code review video in this portfolio is itself a collaboration artifact. It is written for a manager or peer audience, not a grader, and it walks through existing functionality before any enhancement is proposed. It’s basically the same little sequence I would use to bring a real team up to speed, before touching shared code or making changes.

Communicating with Stakeholders

Technical communication has to flex to its audience. The narratives here are aimed at people who understand software broadly, but they might not know Angular or MongoDB internals. That’s a different kind of skill than explaining, say, a trade-off especially when you have to talk to a product manager or a client. In CS 320 (Software Testing, Automation, and Quality Assurance), I practiced translating defect reports into plain-language risk statements for non-technical stakeholders, and I used that same approach again here when I justify why increasing password- hashing iterations from 1,000 to 100,000 matters even if someone has never heard of PBKDF2.

Iterative Testing

CS 320 (Software Testing, Automation, and Quality Assurance) is where iterative, test-driven validation became a habit not really an afterthought. In that course I made a JUnit test suite for a contact management service ContactServiceTest.java , covering more than just the happy path (adding, updating, and deleting a contact) , but also those edge cases that tend to mess things up: duplicate ID rejection, null-input handling, and updates against an ID that doesn’t exist. Honestly, writing tests that confirm an IllegalArgumentException gets thrown for invalid input, not only checking that valid input works, is the same instinct I dragged into this capstone’s validation logic too. Like, the Mongoose schema in Enhancement Three, rejects negative prices and malformed emails at the data layer, for the same reason, catch the bad stuff early and in an explicit way instead of assuming the layer above already took care of it.

Data Structures and Algorithms

Enhancement Two is probably the clearest evidence of this strength. You get a Map based O(1) cache, a binary search over a sorted index, and a generic comparator-based sort that correctly handles string, numeric, and date fields in a single function. These aren’t just abstract exercises. CS 260 (Data Structures and Algorithms) gave me the vocabulary to reason about Big-O complexity, but applying that logic to a real price filter and a real search box where the wrong data type choice can quietly produce the wrong sort order is what made the trade-offs feel concrete instead of theoretical.

Software Engineering and Database

Enhancement One shows software design maturity moving a dual-rendering setup (Express/Handlebars and then a separate Angular SPA) into a single lazy-loaded Angular front end is basically the same class of architectural consolidation that production teams do when technical debt starts piling up. CS 340 (Client/Server Development) is where I first worked on that separation deliberately, not just by accident: for Grazioso Salvare, a rescue-animal organization, I built a MongoDB-backed CRUD module plus a dashboard. I split the database transaction logic from the dashboard’s presentation layer so the CRUD module could be reused behind a different front end without edits. Enhancement Three carries that same instinct further in the Travlr Getaways database layer redesigning the Mongoose schema, adding compound indexes, and migrating a price field from String to Number once I realized the original type choice was quietly sabotaging numeric sorting a bit upstream in the Angular layer. Watching the schema-to-algorithm connection play out in this capstone; like a database decision turning into a sorting bug two layers away. That was the clearest version yet of the lesson CS 340 taught me: where you draw the boundary between a database layer and the code that consumes it is what ultimately decides how much of the system breaks when one side changes.

Security

A security mindset threads through all three enhancements, rather than living in just one place. In Enhancement One, removing the Handlebars rendering layer wipes out a server-side template injection surface, straightforward, but also easy to overlook if you only check outcomes. Then Enhancement Two , with its API controller sort allowlist, stops arbitrary field injection into a MongoDB .sort() call so you don’t end up “sorting” whatever an attacker feels like. Enhancement Three tightens authentication directly: the PBKDF2 iterations got raised to align with current NIST guidance, and an account lockout mechanism was added to resist credential-stuffing attempts. CS 305 (Software Security) is where that sort of thinking became a habit, not a checklist. For Artemis Financial , a client handling private financial data, I ran the OWASP Dependency-Check tool on the codebase, worked through the report to split out actual issues from false positives, then patched dependencies and enforced TLS for data in transit. After that I repeated the scan after each fix, just to confirm nothing new had been introduced. That same iterative cycle of scan, patch, re-verify, document is what shaped how I handled hardening the Travlr Getaways authentication system in Enhancement Three, instead of treating the PBKDF2 iteration count as a one-time tweak to bump and forget.

How the Artifacts Fit Together

Each enhancement kinda builds on what the previous one left behind in the architecture. Enhancement One brings the whole front end together into one Angular codebase and sets up typed domain models, which feels neat. Then Enhancement Two sits on that typed foundation and it adds search, sort, and caching logic , but along the way it exposes a price formatting problem that you can trace back to the original database schema. Enhancement Three then fixes that issue at the source by hardening the schema itself and the authentication system around it, so it stops slipping through. After those three, the next enhancements are presented in that same sequence software engineering and design, algorithms and data structures, and databases , so the architectural narrative matches how it was really built.

What follows: a code review video walking through the original artifact, then three enhancement pages, each containing the original code, the enhanced code, the specific skills demonstrated, and a reflective narrative — followed by a course outcomes summary mapping every enhancement back to the five program outcomes.
